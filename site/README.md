# Documentation site

The public site for `@language-lit/material3-expressive`, published at
[m3e.language-lit.com](https://m3e.language-lit.com).

It is a **consumer** of the package, not part of it. It resolves the library
through the published export map only, so a site build failure is a failure a
real application would hit. ADR 0028 records that decision and its boundaries.

## Run it

```bash
npm install          # in the repository root, builds dist/
npm run site:dev     # http://localhost:3000
```

`site/` has its own `package.json` and lockfile; the library gains no dependency
from it. The `predev`/`prebuild` hooks run `scripts/link-self.mjs`, which links
the package into the repository's own `node_modules` under its published name so
the playground examples — which live above `site/` and import the package the way
any consumer does — resolve. The link points at the package root, so the export
map still applies and a deep import into `src/` still fails.

Because the site consumes the built `dist/`, run `npm run build` in the root
after changing library source to see it here.

## Where the content comes from

Nothing about a component is written twice.

| On the site | Comes from |
| --- | --- |
| Which components exist, and their conformance | `docs/component-inventory.json` |
| Component contract prose | `docs/components/*.md` |
| Live demos and the source shown under them | `playground/examples/*.example.tsx` |
| Guides | `docs/GETTING_STARTED.md` and its siblings |
| Version, bundle sizes, browser support | `package.json`, `docs/bundle-budgets.json`, `docs/browser-support.json` |

`npm run check:site` fails when a conformant component has no route or demo,
when a demo exists for something the inventory does not advertise, when the
generated demo registry is stale, or when the site reaches around the export
map. It runs inside `npm run verify`.

## Generated and vendored files

- `site/demos/registry.tsx` — generated by `npm run generate:site-demos`. Do not
  edit by hand.
- `site/public/fonts/` — Material Symbols subsets containing only the icons this
  site renders, in all three families `symbolStyle` can select (outlined,
  rounded, sharp). Fetched by `npm run generate:site-symbols` and committed so
  builds need no network.

  Re-run that script after adding an icon or using a new `symbolStyle`.
  `check:site` fails when the subset is missing an icon the site renders,
  carries one it no longer renders, or lacks a family a `symbolStyle` selects —
  because a glyph with no font behind it silently renders as its ligature text
  ("home" appears as the word "home") and no build or test would catch it.

## Deployment

Vercel, with **Root Directory** set to `site`. The build is a static export
(`output: 'export'`), so the result is plain files under `site/out` and can be
served by anything.

`npm ci` must run in the repository root before the site build so `dist/` exists.
